# Dependency Catalogs, Reporting, And Rollback

Load this reference when `/etrnl-dev-deps` touches multiple manifests, a dependency family, bot PRs, security alerts, or any lockfile.

## Catalog Consolidation

Use one version source of truth when multiple workspace packages depend on the same external package or on a package family that must move together.

For pnpm workspaces:

- Use `pnpm-workspace.yaml` `catalog` for shared default versions.
- Use named `catalogs` for coherent families that move together, such as `react`, `lint`, `test`, `build`, `db`, `auth`, or `payments`.
- Replace repeated manifest literals with `catalog:` for the default catalog or `catalog:<name>` for a named catalog.
- Keep workspace-internal dependencies on the repo's existing `workspace:` convention instead of moving them into external catalogs.
- After catalog edits, run the package manager install so the lockfile is generated by tooling.
- Run Knip or the repo's equivalent catalog check when available; unused catalog entries are stale dependency state.

When no catalog exists:

- If the repo already uses pnpm workspaces and multiple manifests repeat external versions, create the smallest useful catalog entry set instead of updating each manifest independently.
- If the repo uses npm, Yarn, Bun, Cargo, Go, Python, or another ecosystem, use its existing central version surface only when the repo already has one. Do not perform a package-manager migration just to create a catalog.
- If version divergence is intentional, keep it explicit and record the constraint: framework support, peer range, runtime target, temporary migration, or package-specific breakage.

Do not centralize versions when:

- Packages intentionally test multiple major versions.
- Peer dependency ranges must stay broad for a published library.
- A framework adapter, generated client, or runtime package requires a narrower version than the rest of the repo.
- The user asked for a narrow security patch and catalog migration would expand the blast radius.

## Report Contract

Every dependency audit, report, upgrade, or bot-PR pass reports:

- Mode: `audit`, `upgrade`, `bot-pr`, or `report`.
- Package manager and version.
- Dependency files inspected.
- Vulnerability result by severity and affected package.
- Outdated packages grouped by patch, minor, and major.
- Packages changed, removed, or intentionally skipped.
- Related packages checked.
- Catalog changes: added entries, renamed entries, changed manifest references, unused entries removed, and intentional non-catalog exceptions.
- Lockfiles changed.
- Verification commands and pass/fail results.
- Rollback command sequence.
- Residual risks and source-limited gaps.

For security alerts, include the vulnerable range, fixed range, alert source, exploitability notes if available, and why the selected version is the smallest safe fix.

For bot PRs, include PR source, grouped-package rationale, CI status before/after, release-note risks, and any bot-proposed change rejected.

## Upgrade Order

Use the smallest compatible sequence:

1. Baseline install and quality gate when the repo can run it.
2. Security fixes and patch upgrades.
3. Minor upgrades with related packages.
4. Major upgrades only after explicit approval and migration evidence.
5. Dependency removals only after explicit approval unless the user asked for cleanup and Knip/repo evidence proves the dependency is unused.

When several packages share one contract, upgrade them as a bundle:

- React family: `react`, `react-dom`, framework adapter, testing renderer, types.
- TypeScript/lint family: `typescript`, eslint/oxlint/biome tooling, parser, plugins, config packages.
- Build family: bundler, plugin packages, framework plugin, test transform helpers.
- Database/client family: ORM, generated client, migration tool, adapter.
- Auth/payment family: SDK, webhook verifier, adapter, generated types.

## Rollback Contract

Before edits, identify the exact files likely to change:

- Root and workspace `package.json` files.
- Lockfile.
- Catalog file such as `pnpm-workspace.yaml`.
- Override/resolution files.
- Generated clients or config files required by the package manager.

Record the rollback sequence using the repo's normal VCS workflow. For Git worktrees, the default sequence is:

```bash
git diff -- package.json pnpm-workspace.yaml pnpm-lock.yaml
git restore -- package.json pnpm-workspace.yaml pnpm-lock.yaml
<package-manager> install
<failing gate rerun>
```

Adjust filenames for the actual package manager and workspace files. Do not run broad destructive commands such as `git reset --hard` or `git checkout -- .`.

When a single package in a batch fails:

1. Revert only the manifest, catalog, and lockfile changes for that package or package family.
2. Re-run install.
3. Re-run the gate that failed.
4. Continue only with independent packages.
5. Report the skipped package, attempted version, failing command, and next safe version to investigate.
